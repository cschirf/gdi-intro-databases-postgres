What is data modeling?

  • First step in database design and object oriented programming
  • Goal is to create a conceptual model of how data relate to each other
  • Exploratory process that progresses from:
    • Conceptual Models - high level domain models
    • Logical Data Models - describe logical entities and relationships
    • Physical Data Models - used to design internal database schemas

Modeling Real Things

Kinds of data

  • Strings any sequence of characters (letters, numbers, symbols, whitespace), must be quoted
  • Numbers
    • Integers (int) whole numbers, can be negative, there’s an upper limit
    • Doubles / Floating point decimals, details depend on system/language
    • Bigints / Longs whole numbers beyond the limit of regular integers; there’s still an upper limit, though
  • Boolean / Bit true or false; 1 or 0
  • Arrays Ordered lists; can contain anything
  • Objects A collection of key-value pairs

The Great Balancing Act

  • What data do you need? What data is irrelevant?
    Slogging through unnecessary data makes processing slow.
  • What’s related? Can those pieces be grouped together?
    Putting related information together makes it easier to reason about and can potentionally increase processing efficiency.
  • How hard will it be to update an existing schema?
    Data models are rarely static. What are things you’ll most likely need to update in the future and how hard will it be to do that with the model you’re currently considering?

the great balancing act

It’s a balancing act. Databases are notoriously difficult and risky to migrate. So it is good to make deliberate, informed, and rational decisions about what your data needs are now and what they will likely be in the immediate future. Remember, too, that over-architecting can be dangerous.

But there are no crystal balls. At some point, you need to make a decision and roll with it.

Let’s talk about relationships.

Types of Relationships

  • One-to-One A car model is made by one company.
  • One-to-Many One class has many students.
  • Many-to-One Many employees work in one department (or many students are in a single class).
  • Many-to-Many An author can write many books, and a book can be written by many authors.

Things to think about

  • What things (objects) am I dealing with? What does it mean to be each one of those objects?
  • If I split these objects up, what does that mean for related data?
  • Are the objects related? If so, what kind of relationship am I dealing with? What does that mean for related data?
  • Is there an implicit or explicit hierarchy in the data? What does that mean for sub-objects? For related data?
  • Will this allow me to do more things later, or restrict what I can do later?

What are relational dbs and what do they get me?

  • Mature (read: been around for a while) and still widely used
  • Model relationships between data
  • Allow you to put constraints on your data
    (e.g., any order that is placed in the system must belong to an existing user)
  • ACID (Atomicity, Consistency, Isolation, Durability)
  • Usually, a poor choice for a distributed system
  • They use a dialect of SQL (Structured Query Language)

What about NoSQL dbs?

  • There are many different kinds of non-relational databases
    • aggregate databases
      • document stores (e.g., MongoDB)
      • column family (e.g., Cassandra)
      • key-value stores (e.g., Project Voldemort)
    • graph databases (e.g., Neo4j)
  • Ideal for clusters, distributed systems, and/or large-scale data.
  • BASE (Basically Available, Soft state, Eventual consistency)
  • Schema-less, so you might not always have the data you expect
  • Faster development?
  • Range of interface languages

In an RDB

  • In a RDBMS, entity types generally correspond to tables. For example, a user authentication database might contain tables for users, accounts, and authentication tokens.
  • A collection of related entity types and associated helper objects (e.g., functions, stored procedures, triggers) constitute a database.
  • Just like the data we’ve been modelling, the data in tables can be of different types— integers, strings, doubles or decimals, etc.— although most RDBs use different names.

common data types for RDBs

varchar(n)
similar to strings (“character varying”). You can specify the maximum number of characters the field can have, e.g., varchar(128) will give you a string field that allows a maximum of 128 characters.
text, longtext, clob
Used for very long strings. The details will depend on the RDB used, but generally the allowed size of these strings is from 2-4 GB.
bit, boolean
T/F values. In most RDBs, this is represented by 0s and 1s.
int
integer values (whole numbers)
bigint
bigger integer values than what is allowed by int.
Used for primary keys.
double precision, real, double, float, float(n)
Decimal numbers. Some RDBs will let you specify the degree of precision— float(24). Details will vary.
date, time, datetime, timestamp
Details will vary on which RDB you’re using.

Other possible data types

Depending on what RDB you’re using, you may have access to some or all of the following:

  • smallint
  • serial
  • geometric data types (polygon, circle, path)
  • geographic data types (latitudes, longitudes)
  • text vectors (used in NLP)
  • currency, money
  • uuid, guid
  • xml
  • json

A note on IDs

  • In the recipe example, we assigned an id to each ingredient.
  • This id is unique across the entity type or table (e.g., ingredient).
  • The column containing the id is often referred to as the primary key of the table. Oftentimes, the column is set as serial or auto increment, which means that the database will take care of making sure that each new record gets a unique id.
  • Then we can refer to the ingredient by that unique id, instead of entering the name of the ingredient over and over again.
  • This is the most common way to refer to items in relational databases. (Occassionally, you will see people use a composite key.)
  • Why do you think we use ids?
