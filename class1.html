What is data modeling?

  • How are we going to organize our data?
  • Are pieces of data related? If so, how?

Formal defintions

  • Conceptual Models - high level domain models
  • Logical Data Models - describe logical entities and relationships
  • Physical Data Models - used to design internal database schemas

Modeling Real Things

Kinds of data

  • Strings any sequence of characters (letters, numbers, symbols, whitespace), must be quoted
  • Numbers
    • Integers (int) whole numbers, can be negative, there’s an upper limit
    • Doubles / Floating point decimals, details depend on system/language
    • Bigints / Longs whole numbers beyond the limit of regular integers; there’s still an upper limit, though
  • Boolean / Bit true or false; 1 or 0
  • Arrays Ordered lists; can contain anything
  • Objects A collection of key-value pairs

The Great Balancing Act

  • What data do you need? What data is irrelevant?
    Slogging through unnecessary data makes processing slow.
  • What’s related? Can those pieces be grouped together?
    Putting related information together makes it easier to reason about and can potentionally increase processing efficiency.
  • How hard will it be to update an existing schema?
    Data models are rarely static. What are things you’ll most likely need to update in the future and how hard will it be to do that with the model you’re currently considering?

Let’s talk about relationships.

Types of Relationships

  • One-to-One A car model is made by one company.
  • One-to-Many One class has many students.
  • Many-to-One Many employees work in one department (or many students are in a single class).
  • Many-to-Many An author can write many books, and a book can be written by many authors.

Things to think about

  • What things (objects) am I dealing with? What does it mean to be each one of those objects?
  • If I split these objects up, what does that mean for related data?
  • Are the objects related? If so, what kind of relationship am I dealing with? What does that mean for related data?
  • Is there an implicit or explicit hierarchy in the data? What does that mean for sub-objects? For related data?
  • Will this allow me to do more things later, or restrict what I can do later?

What are relational dbs and what do they get me?

  • Mature (read: been around for a while) and still widely used
  • Model relationships between data
  • Allow you to put constraints on your data
    (e.g., any order that is placed in the system must belong to an existing user)
  • ACID (Atomicity, Consistency, Isolation, Durability)
  • Usually, a poor choice for a distributed system
  • They use a dialect of SQL (Structured Query Language)

The Recipe Example

What about NoSQL dbs?

  • There are many different kinds of non-relational databases
    • aggregate databases
      • document stores (e.g., MongoDB)
      • column family (e.g., Cassandra)
      • key-value stores (e.g., Project Voldemort)
    • graph databases (e.g., Neo4j)
  • Ideal for clusters, distributed systems, and/or large-scale data.
  • BASE (Basically Available, Soft state, Eventual consistency)
  • Schema-less, so you might not always have the data you expect
  • Faster development?
  • Range of interface languages

Entity Modeling

  • Entity type - any type of object that we want to store data about
  • Attribute - the data that we want to keep about each entity
  • Primary key - the attribute used to identify a specific entity
  • Composite key - a primary key made up of a unique combination of attributes
